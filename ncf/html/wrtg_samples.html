User Guides

A user guide contains information about how to install, configure, and make use of a hardware device, system, product, or software application, although sometimes installation, configuration, and troubleshooting instructions might be presented in separate documents. User guides might contain conceptual and reference information as well as sets of steps, or procedures, for performing common operations.

The user guide samples here are examples of two methods of presenting technical information: as single PDF documents (or extracts of PDF documents), and as sets of individual HTML files (topics) that comprise the guide. The latter method enables the documentation to be integrated into the help system of an application or software tool and to be read online. Presenting information in a PDF document enables users to download the file for offline viewing or printing.

Programmer's guide sample

This PDF document, Entrust Security Toolkit for Java, is for a developer audience. The guide describes how to use a set of application programming interfaces (APIs) for cryptographic operations in Entrust's Public Key Infrastructure.

At the time I wrote this document, we were using an in-house XML vocabulary that I helped to design. I wrote this sample document in XML and transformed the source to XSL formatting objects (XSL-FO) using XSLT transformation scripts. I was then able to render the XSL-FO source as the final PDF with the RenderX formatting engine. I've included samples of the scripts I used for this task in the source code section of this page.

Administration Guides

An administration guide is really a user guide that's written for a specific audience, the administrator of a computer hardware or software system. An administrator might be responsible for the installation, configuration, and continuing maintenance of hardware components and software, so the content of an administrator guide contains information appropriate to these tasks.

Reference Documentation

Reference documentation provides its users with categorized information that they can refer to easily when it's needed. Such categories include: API descriptions, configuration settings, policy rules, and so on.

Reference documentation samples

Source Code

Much of the work I've done with in-house, open source, and commercial documentation processing tools involved configuration and customization. This meant that I had to write build scripts, DTDs, XSLT transformation scripts, and on one occasion a Java application to control XML source transformation and output file rendering.

The processing tools that needed this attention included an in-house Java application, several releases of the DITA Open Toolkit, stand-alone XML transformation environments, and a commercial content management system (Ixiasoft DITA CMS).
